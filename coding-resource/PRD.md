## 1. 프로젝트 개요

### 1.1 한 줄 정의

**Claude Code 기반의 AI 에이전트가 한국 교회의 반복적 행정 업무(주보 생성, 새신자 관리, 재정 보고, 문서 생성)를 자율적으로 수행하는 워크플로우 자동화 시스템.**

### 1.2 프로젝트의 본질

이 시스템은 SaaS 제품이 아니다. Claude Code의 `workflow.md` 기반 에이전트 자동화 시스템이며, AgenticWorkflow 코드베이스의 **자식 시스템**이다. 부모의 전체 게놈(헌법 3개, 4계층 품질보장, SOT 패턴, Safety Hook, Context Preservation)을 구조적으로 상속받는다.

### 1.3 시장 기회

| 지표 | 수치 | 출처 |
|------|------|------|
| 한국 개신교 교회 수 | 60,000개+ | 한국기독교연합 |
| 핵심 타겟 (100-500인 중형 교회) | 15,000개 | 시장 분석 보고서 |
| 목회자 AI 사용 의향 | 79.9% (2025) | 기독일보 |
| AI 활용 1순위 분야 | 교회 행정 전산화 63.9% | 목회데이터연구소 |
| 경쟁사 중 AI 자동화 보유 | **0개** (5개사 모두 전무) | 시장 분석 보고서 |
| 한국 ChMS 시장 규모 | USD 14.6M (CAGR 3.6%) | FutureMarketInsights |

**결론**: 시장 수요 검증됨(79.9%). 공급 공백(AI 자동화 0개). 진입 타이밍 최적(2026년 = "어떻게 쓸 것인가"의 해).

## 2. 자동화 목적 및 비전

### 2.1 자동화 목적

> **"AI가 행정 업무를 처리하고, 목회자는 말씀과 성도에 집중한다."**

한국 중형 교회(100-500인)의 행정 간사 또는 목사 겸직자가 주당 약 23시간을 행정 업무에 소비한다. 이 시스템은 반복적 행정 업무를 AI 에이전트에게 위임하여, 행정 시간을 **주당 5시간 이하**로 줄이는 것을 목표로 한다.

### 2.2 자동화 대상 업무

| 업무 | 현재 소요 시간 | 자동화 후 | 절감 |
|------|-------------|----------|------|
| 주보 제작 | 2-3시간/주 | 15분 (검토만) | ~90% |
| 새신자 후속 관리 | 3-5시간/주 | 30분 (승인만) | ~85% |
| 재정 보고서 | 4-6시간/월 | 30분 (검토만) | ~90% |
| 교단 보고서 | 8-12시간/년 | 1시간 (검토만) | ~90% |
| 공문/증명서 작성 | 1-2시간/건 | 10분/건 | ~85% |
| 봉사자 배정 | 2-3시간/주 | 20분 (조정만) | ~85% |

### 2.3 비전 — 3가지 계층의 가치

```
Layer 3: 예측적 목회 (미래)
  └─ 이탈 위험 조기 경보, 교인 성장 여정 추적, 사역 건강성 대시보드

Layer 2: 지능형 분석 (Phase 2)
  └─ 출석 트렌드, 헌금 이상 감지, 새신자 코호트 분석, 봉사 참여 인사이트

Layer 1: 반복 업무 자동화 (MVP — 이 PRD의 범위)
  └─ 주보 생성, 새신자 파이프라인, 재정 보고서, 문서 자동 생성
```

### 2.4 비자동화 원칙

다음은 AI가 대체하지 않는다:
- **목회적 판단**: 심방 여부 결정, 성례 승인, 직분 임명
- **신학적 검증**: 설교 내용 확인, 교리 판단
- **개인 관계**: 교인 대면 상담, 위기 돌봄
- **법적 서명**: 기부금영수증 최종 날인, 공문 직인

AI는 "초안 생성 + 데이터 정리 + 패턴 감지"를 하고, **사람이 검토/승인/서명**한다 (Human-in-the-loop 필수).

### 2.5 AI 초안 생성 vs 실제 발송/전달 구분

> **MVP 범위에서 AI가 수행하는 것은 "메시지 초안 생성"까지이다. 실제 발송은 Phase 2 또는 수동 처리.**

| 작업 | AI 수행 (MVP) | 사람 수행 (수동/Phase 2) |
|------|-------------|----------------------|
| 환영 메시지 | 초안 텍스트 생성 (Markdown) | 카카오톡/문자로 실제 발송 |
| 심방 알림 | 알림 내용 + 대상자 목록 생성 | 담당자가 직접 연락 |
| 봉사자 배정 안내 | 배정 초안 + 연락처 정리 | 담당자가 직접 전달 |
| 재정 보고서 | 보고서 문서 자동 생성 | 인쇄/배포는 사람이 수행 |

> **근거**: 카카오/문자 발송은 API 연동(Phase 2) 또는 수동 전달로 처리. MVP에서 "자동 발송"을 약속하지 않는다.

## 3. 타겟 사용자 및 페르소나

### 3.1 1차 사용자: 행정 간사 (김미영, 42세)

- **역할**: 중형 교회(250인) 행정 간사 (유일한 전임 행정 직원)
- **기술 수준**: 한글/엑셀 능숙, CLI 경험 없음, 스마트폰 일상 사용
- **Pain Point**: 매주 주보 제작 3시간, 월말 재정 보고 6시간, 새신자 관리 엑셀로 수동 추적, 교단 보고서 연 2회 12시간
- **니즈**: "반복 업무를 줄여서 교인 상담에 시간을 쓰고 싶다"
- **기술 제약**: Claude Code CLI를 직접 사용할 수 없음 → **IT 자원봉사자 또는 자연어 추상화 계층 필수**

### 3.2 2차 사용자: 담임 목사 (이성훈, 55세)

- **역할**: 중형 교회 담임 목사 (행정도 겸직)
- **기술 수준**: 스마트폰 기본 사용, 엑셀 미숙, AI 관심은 있으나 직접 사용 경험 없음
- **Pain Point**: "행정에 빼앗기는 시간 때문에 설교 준비와 심방이 부족하다"
- **니즈**: "AI가 행정을 알아서 처리해주고, 나는 승인만 하고 싶다"
- **의사결정**: 시스템 도입 최종 결정권자

### 3.3 3차 사용자: IT 자원봉사자 (박준호, 29세)

- **역할**: 교회 청년부 IT 자원봉사자
- **기술 수준**: 개발자 또는 IT 전공, CLI 능숙, Git 기본 이해
- **Pain Point**: 교회 시스템 설치/유지보수를 주말에 봉사로 수행
- **니즈**: "설치가 간단하고, 유지보수가 거의 필요 없는 시스템"
- **역할**: 시스템 설치, 초기 설정, 문제 해결 1차 대응

### 3.4 4차 수혜자: 장로/재정 담당 집사

- **관심사**: 재정 투명성, 헌금 정확한 집계, 예산 대비 실적 추적
- **니즈**: 월간 재정 보고서 자동 수신, 기부금 영수증 정확한 발급

## 4. 사용자 스토리

### 4.1 핵심 사용자 스토리 (MVP)

#### US-01: 주보 자동 생성

> **AS** 행정 간사,
> **I WANT** 이번 주 설교 정보와 공지사항을 입력하면 주보가 자동으로 만들어지기를,
> **SO THAT** 매주 3시간의 주보 제작 시간을 15분(검토)으로 줄일 수 있다.

**수용 기준**:
- 설교 제목, 본문, 찬송가, 공지사항이 포함된 주보 Markdown 자동 생성
- 생일/기념일 교인이 자동으로 포함됨 (`members.yaml` 참조)
- 이전 주보 양식을 스캔하면 동일 레이아웃 템플릿으로 자동 변환 (scan-and-replicate)
- 담당자가 검토 후 최종 승인 (Human-in-the-loop)

#### US-02: 새신자 케어 자동 파이프라인

> **AS** 담임 목사,
> **I WANT** 새신자가 등록되면 환영 → 재방문 확인 → 소그룹 연결까지 자동으로 진행되기를,
> **SO THAT** 새신자 이탈률을 줄이고, 수동 관리 부담 없이 정착을 도울 수 있다.

**수용 기준**:
- 새신자 등록 시 환영 메시지 초안 자동 생성 (담임 목사 명의)
- 2주 후 재방문 미확인 시 담당자에게 알림 생성
- 정착 단계(first_visit → attending → small_group → baptism_class → settled) 자동 추적
- 각 단계 전환 시 담당자 승인 필요

#### US-03: 재정 보고서 자동 생성

> **AS** 재정 담당 집사,
> **I WANT** 월말에 헌금 집계와 지출 내역이 자동으로 보고서로 만들어지기를,
> **SO THAT** 수작업 집계 오류를 방지하고, 당회에 정확한 재정 현황을 보고할 수 있다.

**수용 기준**:
- 헌금 항목별(십일조, 감사헌금, 특별헌금 등) 자동 집계
- 지출 항목별(관리비, 인건비, 사역비 등) 자동 분류
- 예산 대비 실적 자동 비교
- 산술 정합성 P1 검증 통과 (항목 합계 = 총계)
- 담당자 + 담임 목사 이중 검토 후 확정

#### US-04: 파일 기반 데이터 입력 (inbox/ 패턴)

> **AS** 행정 간사 (비기술 사용자),
> **I WANT** 엑셀이나 워드 파일을 폴더에 넣으면 시스템이 자동으로 읽고 처리하기를,
> **SO THAT** YAML이나 CLI를 모르더라도 데이터를 시스템에 입력할 수 있다.

**수용 기준**:
- `inbox/documents/` 폴더에 놓은 Excel/CSV/Word/PDF 파일 자동 파싱
- 파싱 결과를 사용자에게 보여주고 확인 받은 후 내부 YAML에 반영
- 지원 포맷: `.xlsx`, `.csv`, `.docx`, `.pdf`, `.hwp`(PDF 변환 안내)
- 파싱 오류 시 원본 파일 보존 + 오류 내용 명시

#### US-05: 이미지 기반 데이터 입력

> **AS** 행정 간사,
> **I WANT** 영수증이나 명함 사진을 폴더에 넣으면 자동으로 내용을 읽어서 처리하기를,
> **SO THAT** 수동 입력 없이 재정 기록이나 새신자 연락처를 등록할 수 있다.

**수용 기준**:
- `inbox/images/` 폴더의 영수증 사진 → 금액, 날짜, 항목 자동 추출 → `finance.yaml` 반영
- `inbox/images/` 폴더의 명함 사진 → 이름, 연락처, 소속 자동 추출 → `newcomers.yaml` 반영
- Claude multimodal 이미지 분석 사용 (외부 OCR 불필요)
- 추출 결과를 사용자에게 보여주고 확인 후 반영 (Human-in-the-loop)

#### US-06: 문서 양식 복제 (Scan-and-Replicate)

> **AS** 행정 간사,
> **I WANT** 기존에 사용하던 주보 양식을 스캔해서 올리면 그 양식대로 자동 생성해주기를,
> **SO THAT** 새 양식을 배울 필요 없이 기존 방식 그대로 자동화할 수 있다.

**수용 기준**:
- `inbox/templates/` 폴더에 양식 이미지/PDF 업로드
- Claude multimodal로 레이아웃 구조 추출 (고정 영역 vs 가변 영역)
- 추출된 구조를 `*-template.yaml`로 저장
- 첫 회 구조 확인 human-in-the-loop → 이후 자동 반복 생성
- 적용 영역: 주보, 예배 순서지, 공문, 기부금 영수증, 세례증서, 회의록, 행사 초청장

### 4.2 확장 사용자 스토리 (Phase 2)

#### US-07: 교단 보고서 자동화

> **AS** 담임 목사,
> **I WANT** 연말에 교단 총회 보고서가 자동으로 생성되기를,
> **SO THAT** 매년 10시간 이상의 수작업 보고서 작성을 1시간 검토로 줄일 수 있다.

> **AS** 구역장,

#### US-09: 기부금 영수증 일괄 생성

> **AS** 재정 담당 집사,

#### F-05: 파일 기반 데이터 허브 (inbox/ 패턴)

**설명**: 비기술 사용자가 YAML/CLI 없이 데이터를 시스템에 입력할 수 있는 파일 드롭 인터페이스.

**디렉토리 구조**:

```
inbox/
├── documents/              ← 구조화된 파일 입력
│   ├── 헌금내역.xlsx          → finance.yaml
│   ├── 새신자등록카드.xlsx     → newcomers.yaml
│   ├── 심방일지.docx          → members.yaml (history)
│   └── 교인명부.csv           → members.yaml
│
├── images/                 ← 이미지 기반 데이터 추출
│   ├── receipt-001.jpg        → finance.yaml (지출)
│   ├── namecard-kim.jpg       → newcomers.yaml
│   └── bulletin-text.jpg      → bulletin-data.yaml
│
└── templates/              ← Scan-and-Replicate 양식
    ├── bulletin-sample.jpg    → bulletin-template.yaml
    ├── receipt-form.jpg       → receipt-template.yaml
    ├── letter-sample.jpg      → letter-template.yaml
    ├── certificate-sample.jpg → certificate-template.yaml
    ├── worship-order.jpg      → worship-template.yaml
    ├── meeting-minutes.jpg    → minutes-template.yaml
    └── invitation-sample.jpg  → invitation-template.yaml
```

**3-Tier 데이터 수집 계층**:

| Tier | 입력 형식 | 처리 방법 | 정확도 |
|------|----------|----------|--------|
| **A: 구조화 파일** | Excel, CSV | `openpyxl`/`pandas` 파싱 | 높음 |
| **B: 문서 파일** | Word, PDF | `python-docx`/Claude Read | 중간-높음 |
| **C: 이미지 파일** | JPG, PNG | Claude multimodal 분석 | 중간 (확인 필수) |

**HWP 처리**: HWP는 바이너리 포맷으로 직접 파싱 불가. 사용자에게 "PDF로 내보내기 후 업로드" 안내. 또는 `pyhwp` 라이브러리 시도 (제한적 지원).

**공통 원칙**: 모든 Tier에서 파싱 결과를 사용자에게 보여주고, **확인 후에만** 내부 YAML에 반영한다 (Human-in-the-loop).

#### F-06: Scan-and-Replicate (문서 양식 복제)

**설명**: 기존 문서 양식을 이미지/PDF로 업로드하면, 레이아웃 구조를 분석하여 템플릿을 생성하고, 이후 데이터 기반으로 동일 양식의 문서를 반복 자동 생성한다.

**동작 흐름**:

```
inbox/templates/{category}-sample.{jpg|pdf}
    ↓
Step 1: Claude Multimodal 분석
  ├─ 레이아웃 구조 추출 (섹션 배치, 컬럼, 스타일 힌트)
  ├─ 고정 영역 식별 (교회명, 로고 위치, 직인 위치)
  └─ 가변 영역 식별 (날짜, 내용, 수신인 등)
    ↓
Step 2: 템플릿 생성
  ├─ {category}-template.yaml (구조 정의)
  └─ {category}-output.md (Markdown 출력 형식)
    ↓
Step 3: 사용자 확인 (Human-in-the-loop)
  ├─ 추출된 구조가 맞는지 확인
  └─ 가변 슬롯이 올바른지 확인
    ↓
Step 4: 반복 자동 생성
  └─ data/*.yaml → 템플릿 채움 → output/{category}/{date}-{name}.md
```

**적용 가능 7개 영역**:

| 영역 | 반복 빈도 | 데이터 연동 | 우선순위 |
|------|----------|-----------|---------|
| **주보** | 매주 | `bulletin-data.yaml` + `schedule.yaml` + `members.yaml` | Tier A (즉시) |
| **기부금 영수증** | 연말 일괄 | `finance.yaml` + `members.yaml` | Tier A (즉시) |
| **예배 순서지** | 매주 | `schedule.yaml` + `bulletin-data.yaml` | Tier A (즉시) |
| **교회 공문** | 월 2-5회 | `members.yaml` + `schedule.yaml` | Tier B |
| **회의록** | 월 1-4회 | `members.yaml` (참석자) | Tier B |
| **세례/입교 증명서** | 연 2-4회 | `members.yaml` (성례 이력) | Tier B |
| **교단 보고서** | 연 1-2회 | 전체 데이터 집계 | Tier C |

## 6. 데이터 아키텍처

### 6.1 설계 원칙

> **모든 데이터는 외부 API 없이, 현재 코드베이스 안의 YAML 파일로 관리한다.**

**YAML 선택 근거**:
1. 코드베이스 일관성: `state.yaml` SOT, `glossary.yaml` 패턴 이미 존재
2. 에이전트 친화성: Claude Read/Edit/Write 도구로 직접 조작
3. PyYAML 검증: `validate_*.py` 패턴으로 구조적 무결성 보장
4. Git diff 가독성: 변경 이력이 사람이 읽을 수 있는 형태
5. 오프라인 작동: 네트워크 장애에 무관
6. 데이터 주권: 교회 로컬에 데이터 보관 (민감 교인 정보 외부 전송 없음)

### 6.2 데이터 스키마 요약

| 파일 | 용도 | 주요 필드 | 검증 규칙 |
|------|------|----------|----------|
| **`members.yaml`** | 교인 명부 | id, name, contact, church(직분/세례/부서), family, history | M1-M6 (ID 유일, 상태값, 날짜 형식, 가족 참조, 통계 정합) |
| **`finance.yaml`** | 재정 데이터 | offerings(항목별), expenses(카테고리별), budget, monthly_summary | F1-F5 (산술 정합, 월별 합산, void 처리, 양수 검증, ID 유일) |
| **`schedule.yaml`** | 예배/행사 일정 | regular_services, special_events, facility_bookings | S1-S5 (날짜 형식, 시간 형식, 공간 충돌, 상태값, ID 유일) |
| **`newcomers.yaml`** | 새신자 추적 | journey_stage, journey_milestones, assigned_to | N1-N5 (ID 유일, 단계 정합, 날짜 형식, 참조 무결, 이관 일관) |
| **`bulletin-data.yaml`** | 주보 소스 | sermon, worship_order, announcements, prayer_requests | B1-B3 (필수 섹션, 발행 번호 연속, 데이터 참조) |
| **`church-glossary.yaml`** | 교회 용어 사전 | korean, english, context | 교회 특수 용어 한-영 매핑 |
| **`church-state.yaml`** | 교회 행정 SOT | church info, data_paths, current states | Orchestrator만 수정 |

### 6.3 데이터 무결성 보장

**3계층 무결성 구조** (AgenticWorkflow 패턴 재사용):

| 계층 | 메커니즘 | 보장 내용 |
|------|---------|----------|
| **1. 쓰기 권한 분리** | SOT 패턴 (절대 기준 2) | `church-state.yaml` = Orchestrator만. 각 데이터 파일 = 담당 에이전트만 |
| **2. P1 결정론적 검증** | `validate_*.py` 패턴 | 산술 정합, ID 유일, 참조 무결, 형식 검증 |
| **3. 원자적 쓰기** | `fcntl.flock()` + tempfile + rename | 파일 손상 방지, 동시 쓰기 충돌 방지 |

### 6.4 민감 데이터 보호

```

```

별도 암호화 백업 + 로컬 보관 원칙. Git 이력에 민감 정보 노출 방지.

**백업/복구 프로토콜** (data-architecture.md §3-2, §3-3):

| 항목 | 방법 |
|------|------|
| **일일 자동 백업** | cron 스크립트로 `data/` 디렉토리 스냅샷 (로컬 별도 경로) |
| **Git 이력 조회** | `git log --oneline data/finance.yaml`로 변경 이력 확인 |
| **특정 시점 복원** | `git checkout [commit-hash] -- data/finance.yaml`로 개별 파일 복원 |
| **전체 복원** | 일일 백업 스냅샷에서 `data/` 디렉토리 전체 복원 |

> **근거**: 재정 데이터 복구 불가 시 법적/신뢰 문제 발생 (§11.1 위험 가정 #2 직결). 복구 프로토콜은 Day-1부터 필수.

### 6.5 규모 한계

| 한계 | 임계값 | 대응 |
|------|--------|------|
| 교인 수 | ~1,000명까지 단일 파일 | 초과 시 알파벳 분리 (members-A.yaml) |
| 재정 이력 | 연간 100-200건 | 연도별 분리 (finance-2026.yaml) |
| 동시 접근 | 에이전트 순차 쓰기 | SOT 패턴으로 단일 쓰기 지점 |
| 복잡 쿼리 | 직접 쿼리 불가 | 에이전트가 Python으로 필터링 |

**현실적 커버 가능 규모**: **500인 이하 교회** (한국 교회의 약 95%).

## 7. 확장성 아키텍처

### 7.1 핵심 원리: "부모-자식" 확장 패턴

새 기능 = 새 코드 추가가 아니라, **새 자식 워크플로우 탄생**. `workflow-generator` 스킬이 부모의 전체 DNA(4계층 QA, SOT 패턴, Safety Hook)를 자동 상속하는 자식 `workflow.md`를 생성한다.

### 7.2 3-Layer Module Pattern

모든 기능은 동일한 3계층 구조를 따른다:

```
Layer 1 — 워크플로우 (설계도): workflows/{feature}.md
    ↓ 정의
Layer 2 — 에이전트 (실행자): .claude/agents/{specialist}.md
    ↓ 사용
Layer 3 — 데이터 (상태): data/{domain}.yaml (SOT)
```

### 7.3 기능 추가 프로토콜 (5단계)

```
Step 1 — 분류: 기존 워크플로우 확장 vs 새 독립 기능?
         판단 기준: 동일 트리거(매주 실행) → 기존에 단계 추가 / 독립 트리거(월간·온디맨드) → 새 workflow.md
Step 2 — 데이터 의존성 매핑: 어떤 공유 SOT를 읽는가? + 새 기능 전용 SOT 파일 설계
Step 3 — 에이전트 재사용 판단: 기존 에이전트 조합 가능 → 재사용 / 새 전문 역량 필요 → 신규 에이전트 생성
         신규 에이전트는 반드시 Inherited DNA 조항(품질 절대 기준 자동 상속) 포함 필수
Step 4 — workflow-generator로 workflow.md 자동 생성
         → DNA Inheritance P1 검증 실행: validate_workflow.py (W1-W8 구조적 무결성 확인)
Step 5 — Hook 등록 (선택): 검증 Hook 필요 시 settings.json 추가
```

### 7.4 공유 SOT 참조 패턴

```
         ┌────────────────────────────────┐
         │       공유 SOT (읽기 전용)        │
         │  members.yaml  finance.yaml     │
         │  schedule.yaml  newcomers.yaml  │
         └────────────────────────────────┘
                    ↑ 읽기 전용
     ┌──────────────┼──────────────┐
     │              │              │
┌──────────┐  ┌──────────┐  ┌──────────┐
│ bulletin │  │newcomer  │  │volunteer │
│ workflow │  │ workflow │  │ workflow │
│ state    │  │ state    │  │ state    │
└──────────┘  └──────────┘  └──────────┘
  (기능별 독립 SOT — 쓰기 충돌 없음)
```

**공유 SOT 참조 3규칙** (extensibility-architecture.md §4.1):

1. **공유 SOT**(members.yaml 등): Orchestrator만 쓰기, 모든 워크플로우가 읽기 전용으로 참조
2. **기능별 SOT**(bulletin-state.yaml 등): 해당 워크플로우의 Orchestrator만 쓰기
3. **교차 워크플로우 데이터**: 공유 SOT를 통해서만 교환. **워크플로우 간 직접 SOT 참조 금지.**

### 7.5 IT 자원봉사자 기능 추가 경로

**자연어 요청**: "소그룹 출석 관리 워크플로우 만들어줘" → `workflow-generator` 스킬 자동 생성 → DNA Inheritance 자동 포함 → 즉시 실행 가능.

**템플릿 복사**: 기존 `workflow.md`를 복사 → 내용 교체 → Inherited DNA 유지 → `validate_workflow.py`로 검증.

### 7.6 스키마 확장 규칙 (하위 호환성)

공유 SOT(`members.yaml` 등)에 새 기능을 위한 필드를 추가할 때, 기존 워크플로우가 깨지지 않도록 아래 원칙을 **반드시** 준수한다:

1. **기존 필드 삭제 또는 이름 변경 금지** — 기존 워크플로우가 참조하는 필드를 제거하면 즉시 파손
2. **새 필드는 항상 선택적(optional)** — 기존 워크플로우는 새 필드를 무시해도 정상 동작
3. **`.get()` 방어 패턴 필수** — `member.get('volunteer', {})` 형태로 필드 존재 여부를 검사

### 7.7 확장성 보장 4원칙

| 원칙 | 메커니즘 | 확장 시 이점 |
|------|---------|-----------|
| **DNA 유전** | 모든 workflow.md에 Inherited DNA 섹션 필수 | 새 기능이 자동으로 4계층 품질보장 상속 |
| **SOT 분리** | 기능별 독립 SOT + 공유 SOT 읽기 전용 참조 | 기능 추가가 기존 데이터를 오염시키지 않음 |
| **에이전트 조합** | 기존 에이전트 재사용 + 필요 시 신규 추가 | 전문성 축적 — 새 기능이 기존 품질 인프라 활용 |
| **Hook 격리** | `if test -f` 패턴으로 Hook 선택적 활성화 | Hook 추가/제거가 기존 시스템에 영향 없음 |

> **확장 시 알려진 제약**: (1) 공유 SOT 쓰기 병목 — `members.yaml` 동시 갱신 불가, 워크플로우 간 조율 프로토콜 필요. (2) 에이전트 수 증가에 따른 컨텍스트 관리 복잡도. (3) 데이터 스키마 비대화 위험 — 정기 스키마 검토 + §7.6 선택적 필드 원칙. (4) 비전문 IT 자원봉사자의 Verification 기준 작성 어려움 — workflow-generator가 기준 예시를 제안.

## 9. 기술 아키텍처 제약

### 9.1 절대 제약

| 제약 | 근거 |
|------|------|
| **외부 API 의존 금지** | 사용자 명시적 지시: "모든 데이터는 코드베이스 안에서 해결" |
| **Claude Code 구독 계정 단일 실행 환경** | 모든 AI 기능은 Claude Code 구독 계정의 내장 도구(Read, Write, Edit, Bash, Task, multimodal 이미지 분석)로 동작. 별도 API 키·외부 AI 서비스 불필요 |

| **SOT 단일 쓰기 원칙** | 각 데이터 파일에 쓰기 권한 에이전트 1개만 지정 |
| **재정 데이터 Autopilot 금지** | 높은 위험 업무는 항상 수동 승인 |

| **실패 시 대응** | 대안 입력 방식 설계 (예: 이메일 전송→자동 수집, 간단 웹 폼) |
