### 분석 기준

### ✅ 전체 합의 항목 (4/4 동의)

**1. 주보 자동 생성 워크플로우**
- Market: ✓ (미충족 니즈 #1로 명시 — "주보 자동 생성 불가")
- User: ✓ (필수 3기능 #1 — "없으면 망함")
- Tech: ✓ (MVP 4기능 #1 — "복잡도 낮음, 즉시 구현 가능")
- Business: ✓ (필수 기능 3가지 #2 — "주보 초안 자동 생성")
- **근거**: 유일하게 4개 관점이 낮은 기술 복잡도 + 높은 사용자 수요 + 즉각적 ROI를 동시에 확인한 기능. 4개 관점의 "이것이 먼저"가 정확히 일치.

**2. 새신자 등록-추적-심방 파이프라인**
- Market: ✓ ("새신자 케어 자동화 부재" 미충족 니즈 명시)
- User: ✓ (필수 3기능 #2 — "등록-추적-심방 파이프라인")
- Tech: ✓ (MVP 4기능 #3 — "새신자 환영 프로세스, 복잡도 낮음")
- Business: ✓ (필수 기능 #3 — "새신자/교인 연락 자동화")
- **근거**: 교회 고유 Pain Point (교적 이탈 → 교회 약화)와 직결. 기술적으로도 Gmail/Calendar MCP 기존 자산으로 구현 가능.

**3. Human-in-the-loop 명문화**
- Market: ✓ (명시적 언급은 없으나 "AI 채택 가속" 맥락에서 신뢰 장벽 암시)
- User: ✓ ("헌금/재정 데이터 인간 검토 게이트 (오류 책임 UX)" 필수로 명시)
- Tech: ✓ ("Human-in-the-loop 필수" — 아키텍처 원칙 #3)
- Business: ✓ ("'AI가 초안·제안, 목사님이 확인·승인' — Human-in-the-loop 명문화" 절대 빼면 안 됨)
- **근거**: 한국 교회 문화의 보수성 + 재정 오류에 대한 법적·신뢰 리스크. 4개 관점이 이유는 달라도 결론이 동일.

**4. 자연어 진입점 (Claude Code 진입 장벽 완화)**
- Market: ✓ ("대화형 교회 행정 인터페이스" 차별화 포인트 #2)
- User: ✓ ("학습 없이 쓸 수 있는 자연어 진입점" 절대 포함 필수)
- Tech: ✓ (리스크 R3 — "비기술 사용자의 Claude Code 진입 장벽"이 주요 리스크)
- Business: ✓ (도입 장벽 ★★★★★ 치명적 — "CLI는 비기술자에게 불가")
- **근거**: 접근성 문제를 4개 관점이 인식하되 해법 합의 아직 없음. "인식은 전체 합의, 해법은 충돌"이므로 진입 장벽 자체의 존재는 GREEN.

**7. 교단별 보고 양식 내장**

**8. 소형교회 무료 티어**

## STEP 2: 충돌 지점 명시화 (Conflict Mapping)

---

### 패턴 1: "사용자가 원하는" vs "기술이 할 수 있는"

**충돌 A — 카카오 연동 시점**
- **User 입장**: "한국 교회 소통 99% 카카오 — 이게 없으면 일상 업무 자동화 불가"
- **Tech 입장**: "카카오 API MVP 포함 금지 — 공식 API 제한, 비즈니스 채널 승인 프로세스, 안정성 위험"
- **모더레이터 판정**: Tech가 맞다. 카카오 비즈니스 채널 승인은 수주~수개월 소요. MVP에서 실패하면 전체 신뢰도 타격. **Yellow Zone — Phase 1 완료 후 즉시 Phase 2로 진입하는 로드맵 약속.**

**충돌 B — 재정 관리 복잡도**
- **User 입장**: "헌금 집계 + 재정 보고서가 없으면 현재 Pain Point의 핵심을 해결 못함"
- **Tech 입장**: "재정 관리 복잡도 높음 — 회계 정확성 요구, 외부 데이터 파편화, MVP 범위 초과"
- **모더레이터 판정**: 범위 협상이 가능하다. Tech의 "월간 출석/헌금 보고서 (MVP #4, 복잡도 중간)"는 User가 원하는 재정 기능의 하위 집합. **Yellow Zone — 단, Google Sheets 연동(MVP 전제) 확보 후에만 진행.**

---

### 패턴 2: "시장이 요구하는" vs "기술 타임라인"

**충돌 C — AI 모멘텀 vs 개발 현실**
- **Market 입장**: "2026년이 골든 윈도우 — 지금이 시장 진입 타이밍"
- **Tech 입장**: "데이터 소스 파편화(R1)가 가장 치명적 — 네이버/카카오/엑셀 혼재 문제가 해결 전에는 자동화 불가"
- **모더레이터 판정**: 시장 타이밍 압력이 기술 리스크를 눈 감게 만들 수 있다. R1(데이터 파편화)이 해결되지 않으면 어떤 기능도 제대로 작동하지 않는다. **Google Workspace First 전략(Tech 아키텍처 원칙 #1)을 Market도 수용해야 함.**

**충돌 D — "Cross-system 통합 오케스트레이션" 차별화**
- **Market 입장**: "경쟁사 모두 AI 전무 — Cross-system 통합이 유일 차별화"
- **Tech 입장**: "단일 통합 워크플로우 금지, 교회 관리 시스템 직접 DB 연동 금지"
- **모더레이터 판정**: Market의 "Cross-system 통합"과 Tech의 "워크플로우 독립성"은 실현 방식에서 충돌. 해법: Google Workspace(Sheets, Calendar, Gmail)를 허브로 삼아 데이터 통합 — 직접 DB 연동 없이도 Cross-system 효과 달성 가능. **Green Zone으로 이동 가능(조건부).**

---

### 패턴 3: "사용자 성공" vs "비즈니스 지표"

**충돌 E — 타겟 교회 규모**
- **User 입장**: "한국 교회 86%가 직원 1~4명 소규모 → 자동화 수혜 최대 → 무료 티어로 접근"
- **Business 입장**: "200-500명 중형 교회 타겟 → 행정 간사 1명 + 목사 겸직이 고통 최대 → 유료 모델"
- **모더레이터 판정**: 이 충돌은 고객 세그먼트 철학 차이다. 둘 다 맞다. **하이브리드 전략 필요**: 소형교회 무료(입소문) + 중형교회 유료(매출). 단, MVP는 중형교회 Pain Point 해결 중심으로 개발 → 소형교회는 기능 subset 무료 제공.

**충돌 F — 성공 메트릭**
- **User 정의 성공**: 목사 번아웃 해소, 행정 시간 주 5시간 이하, 새신자 이탈 감소
- **Business 정의 성공**: 월 구독 매출, CAC 회수 기간, NPS
- **모더레이터 판정**: 단기(3개월) 지표는 User 중심(시간 절감, 워크플로우 완료율), 중기(12개월) 지표는 Business 중심(구독 전환율, MRR). KPI 이중 트랙 필요.

---

### 패턴 4: "완벽함" vs "빠름"

**충돌 G — MVP 범위의 경계**
- **더 포함 원하는 측**: User(카카오 연동, 교단 양식), Market(Cross-system 통합)
- **더 빼고 싶은 측**: Tech(재정, 봉사자 배정, 카카오, 소그룹, 교인 DB)
- **Business 위치**: Tech와 대체로 동의하나 재정은 MUST HAVE
- **모더레이터 판정**: Tech의 범위 판단이 현실적이다. 단, 재정의 경우 "완전한 재정 관리" vs "헌금 집계 + 월간 보고서"의 스코프 협상이 필요. 후자는 MVP 가능.

**충돌 H — Autopilot 적용 범위**
- **Tech 입장**: "Autopilot 전면 적용 금지"
- **상충**: AgenticWorkflow 시스템 자체는 Autopilot 지원
- **모더레이터 판정**: 교회 행정의 재정/헌금 데이터는 Human-in-the-loop 필수 → Autopilot은 "주보 초안 생성" 같은 저위험 작업에만 적용. **위험도 기반 Autopilot 범위 정의 필요.**

---

### 패턴 5: "기술 접근성" — 모두가 우려하는 핵심 리스크

**충돌 I — Claude Code CLI 진입 장벽 해법**

| 관점 | 인식 | 제안 해법 |
|------|------|----------|
| Market | "대화형 교회 행정 인터페이스"로 마케팅 | 자연어 인터페이스 강조 |
| User | "학습 없이 쓸 수 있는 자연어 진입점" | UX 단순화 |
| Tech | 리스크 R3 — "IT 담당 교인 자원봉사 의존 구조" | 설치 가이드, 원클릭 설정 |
| Business | "CLI는 비기술자에게 불가 → 치명적 진입 장벽" | IT 담당 교인 에코시스템 |

- **모더레이터 판정**: 4개 관점 모두 "이게 문제다"는 인식은 동일하나, 해법에 합의 없음. Claude Code CLI 자체를 숨기는 **추상화 계층**(웹 인터페이스, 설치 스크립트, IT 자원봉사자 가이드 패키지)이 필요. 이것이 PRD의 "UX 레이어" 핵심 과제.

     - 교회 IT 자원봉사자 커뮤니티 육성 (에코시스템)
   리스크 소유자: Tech + Business

2️⃣ "헌금/재정 자동화가 법적·신뢰 장벽을 통과할 수 있다"
   영향도: 높음 (재정 오류 = 교회 신뢰 붕괴)
   발생확률: 중간 (Human-in-the-loop으로 완화 가능하나 오류 시 책임 소재 불명확)
   검증 방법:
     - 법률 검토: 개인정보보호법 + 교회 회계 처리 기준 적법성
     - 파일럿: 3개 교회 재정 데이터 처리 후 교회 측 감사 결과 확인
   대응책:
     - "AI 제안, 인간 승인" 워크플로우 100% 강제
     - 오류 책임 면책 조항 서비스 약관 명시
     - 재정 기능 별도 보험 검토
   리스크 소유자: User + Business

3️⃣ "Google Workspace가 교회의 기본 데이터 허브로 사용된다"
   영향도: 높음 (Tech 아키텍처 전체의 전제)
   발생확률: 중간 (소형교회는 네이버 카페, 엑셀, 카카오톡 혼재 사용 다수)
   검증 방법:
     - 10개 교회 현재 사용 툴 스택 인터뷰
     - Google Workspace 미사용 교회의 마이그레이션 의향 조사
   대응책:
     - Google Workspace Free Tier 도입 가이드 번들링
     - 네이버/카카오 데이터 → Google Sheets 마이그레이션 툴 선제 개발
     - 마이그레이션 거부 교회는 Phase 2 타겟으로 분류
   리스크 소유자: Tech

4️⃣ "한국 개신교 목사가 AI를 행정 도구로 기꺼이 채택한다"
   영향도: 높음 (채택율 0%면 시장 없음)

     - 목사 기술 수용성 스펙트럼별 온보딩 차별화
     - 교회 내 성공 사례 간증 마케팅 (신뢰 전이)
   리스크 소유자: Market + User

5️⃣ "경쟁사가 AI를 도입하기 전 시장을 선점할 수 있다"
   영향도: 중간 (타이밍 모멘텀 가정)
   발생확률: 중간 (지저스온/치윰 등이 AI 추가 가능 — 자금력 있음)
   검증 방법:

     - 워크플로우 커스터마이징 생태계로 경쟁 진입 장벽 강화
   리스크 소유자: Market + Business

## 모더레이터 최종 소견

4개 관점의 조사 결과는 **"무엇을 만드는가"에는 강하게 합의하고, "어떻게 만드는가"에서 갈린다.**

핵심 긴장은 세 쌍이다:

**긴장 1 — 완벽함 vs 빠름**: User와 Market은 더 많이 원하고, Tech는 더 적게 만들자고 한다. 해법은 Tech가 옳다. **MVP 범위 절제**가 생존의 조건.

**긴장 2 — 소형 vs 중형 타겟**: User의 소형교회 감성과 Business의 중형교회 현실. 해법은 하이브리드 — **중형교회 Pain Point로 제품 설계, 소형교회 무료로 채널 확보.**

**긴장 3 — 카카오 지금 vs 나중**: 사실상의 최대 갈등. 해법은 Tech의 손 — **카카오는 Phase 2, Google Workspace로 MVP 데이터 허브 확보.**

**가장 중요한 단 하나의 우선순위**: 기술 진입 장벽(위험 가정 #1)을 MVP보다 먼저 해결해야 한다. 아무리 좋은 기능도 목사님이 설치할 수 없으면 존재하지 않는 것과 같다.
