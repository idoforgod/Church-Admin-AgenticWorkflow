## Section 1: Entity-Relationship Model

### 1.1 Overview of Core Data Domains

The system operates on 6 core data domains stored as YAML files, plus a central SOT (`church-state.yaml`). All data lives locally in the codebase — no external APIs. The Orchestrator is the sole writer to `church-state.yaml`; each domain file has a designated writer agent.

```
church-state.yaml (SOT — Orchestrator only)
    ├── data/members.yaml          (교인 명부)
    ├── data/finance.yaml          (재정 데이터)
    ├── data/schedule.yaml         (예배/행사 일정)
    ├── data/newcomers.yaml        (새신자 추적)
    ├── data/bulletin-data.yaml    (주보 소스 데이터)
    └── data/church-glossary.yaml  (교회 용어 사전)
```

#### Journey Stage Transition Rules

```
first_visit → attending → small_group → baptism_class → baptized → settled
```

- Each stage requires `completed: true` on all previous milestones before the current stage can be set
- Stage transitions require human approval (`(human)` checkpoint)
- Upon `settled`, the newcomer record links to `members.yaml` via `settled_as_member` field

### 1.9 Cross-Entity Dependencies

```
members.yaml ─────────────────────────────────────────┐
  │                                                    │
  │ family.family_id ←──→ family.family_id (同家族)     │
  │                                                    │
  ├── referred_by ← newcomers.yaml (초청 교인 참조)      │
  ├── assigned_to ← newcomers.yaml (담당 목양자 참조)     │
  ├── settled_as_member ← newcomers.yaml (정착 후 링크)  │
  │                                                    │
  ├── member_id ← bulletin-data.yaml (생일 교인)        │
  ├── family_id ← bulletin-data.yaml (결혼기념일)        │
  │                                                    │
  └── approved_by ← finance.yaml (지출 승인자)           │
                                                       │
schedule.yaml ──→ bulletin-data.yaml (예배 순서 참조)    │
                                                       │
newcomers.yaml ──→ members.yaml (정착 시 이관)          │
                                                       │
church-state.yaml ──→ all data files (경로 참조)        │
                                                       │
church-glossary.yaml ──→ all agents (용어 참조)         ┘
```

**Key Referential Integrity Rules**:
1. `newcomers[].referred_by` → must exist in `members[].id` or be null
2. `newcomers[].assigned_to` → must exist in `members[].id`
3. `newcomers[].settled_as_member` → must exist in `members[].id` when set
4. `bulletin-data.celebrations.birthday[].member_id` → must exist in `members[].id`
5. `bulletin-data.celebrations.wedding_anniversary[].family_id` → must exist in `members[].family.family_id`
6. Same `family_id` must have ≥ 2 members
7. `finance.expenses[].approved_by` → must be a role with approval authority

## Section 4: Human-in-the-Loop Architecture

### 4.1 Risk Classification Matrix

| Risk Level | Operations | Review Type | Autopilot Eligible | Rationale |
|------------|-----------|-------------|---------------------|-----------|
| **HIGH** | Finance recording | Double-review (finance deacon + senior pastor) | **Never** | Legal liability (Tax law §80①5), trust collapse on error |
| **HIGH** | Tax donation receipts | Double-review (finance deacon + senior pastor) | **Never** | Legal document with tax implications |
| **HIGH** | Denominational reports | Double-review (admin secretary + senior pastor) | **Never** | Submitted to denominational headquarters |
| **HIGH** | Member status changes (transfer, deceased) | Double-review (admin + pastor) | **Never** | Permanent, irreversible changes to church register |
| **HIGH** | Newcomer → Member migration | Double-review (newcomer tracker + pastor) | **Never** | Permanent membership status change |
| **MEDIUM** | Bulletin content | Single-review (admin secretary or pastor) | Low-risk portions only | Public-facing document, errors visible to congregation |
| **MEDIUM** | Newcomer stage transitions | Single-review (assigned caregiver) | **No** | Pastoral judgment required for relationship stages |
| **MEDIUM** | Official documents (transfer cert, baptism cert) | Single-review (admin secretary) | **No** | Requires physical seal/signature |
| **MEDIUM** | Volunteer assignments | Single-review (department head) | **No** | People-sensitive scheduling |
| **MEDIUM** | Event/schedule changes | Single-review (admin secretary) | **No** | Facility conflicts possible |
| **LOW** | Birthday/anniversary lookups | Auto-approved | **Yes** | Read-only query from members.yaml |
| **LOW** | Attendance statistics | Auto-approved | **Yes** | Read-only aggregation |
| **LOW** | Glossary term additions | Auto-approved | **Yes** | Append-only, non-destructive |
| **LOW** | Bulletin draft generation (pre-review) | Auto-approved (draft only) | **Yes** | Draft stage only, review gate before finalization |
| **LOW** | Prayer request categorization | Auto-approved (categorization only) | **Yes** | Auto-classification, human reviews list |

### 4.2 Approval Flow Diagrams

**High Risk (Finance Example)**:
```
[AI Agent generates report]
    → [Finance Deacon reviews arithmetic + categories]
        → APPROVE → [Senior Pastor reviews totals + narrative]
            → APPROVE → [Report finalized + archived]
            → REJECT → [Return to AI with feedback]
        → REJECT → [Return to AI with corrections]
```

**Medium Risk (Bulletin Example)**:
```
[AI Agent generates bulletin draft]
    → [Admin Secretary reviews content + layout]
        → APPROVE → [Bulletin finalized]
        → REJECT → [AI regenerates with feedback]
```

**Low Risk (Query Example)**:
```
[User asks "이번 주 생일 교인 목록"]
    → [AI queries members.yaml]
        → [Results returned immediately — no approval needed]
```

## Section 5: Validation Rule Catalog

### 5.1 Members Validation Rules (M1-M6)

| Rule | Check Condition | Severity | Implementation |
|------|----------------|----------|----------------|
| **M1** | All `id` values are unique across all member records | FAIL | `len(ids) == len(set(ids))` |
| **M2** | `status` is one of: `active`, `inactive`, `transferred`, `deceased` | FAIL | `status in VALID_STATUSES` |
| **M3** | `birth_date`, `registration_date`, `baptism_date` match YYYY-MM-DD format and are valid calendar dates in the past | FAIL | `datetime.strptime(date, "%Y-%m-%d") <= today` |
| **M4** | Every `family_id` that is not null has at least 2 members sharing the same `family_id` | WARNING | `count(family_id) >= 2` for each non-null family_id |
| **M5** | `role` is one of: `목사`, `장로`, `집사`, `권사`, `성도`, or null | FAIL | `role in VALID_ROLES or role is None` |
| **M6** | `_stats.total_active` equals the actual count of members with `status == "active"` | FAIL | `computed_count == _stats.total_active` |

### 5.2 Finance Validation Rules (F1-F5)

| Rule | Check Condition | Severity | Implementation |
|------|----------------|----------|----------------|
| **F1** | For each offering: `total == sum(items[].amount)`, tolerance < 1 KRW | FAIL | `abs(total - sum(amounts)) < 1` |
| **F2** | For each month in `monthly_summary`: `total_income == sum(non-void offerings)` and `total_expense == sum(non-void expenses)` and `balance == total_income - total_expense` | FAIL | Arithmetic cross-check across all entries for month |
| **F3** | Entries with `void: true` are excluded from all `monthly_summary` calculations | FAIL | Filter `void != true` before aggregation |
| **F4** | All `amount` values are positive integers (> 0) | FAIL | `isinstance(amount, int) and amount > 0` |
| **F5** | All `id` values are unique within their series (`OFF-*` unique, `EXP-*` unique) | FAIL | `len(off_ids) == len(set(off_ids))` |

### 5.3 Schedule Validation Rules (S1-S5)

| Rule | Check Condition | Severity | Implementation |
|------|----------------|----------|----------------|
| **S1** | All `date` fields match YYYY-MM-DD format and are valid calendar dates | FAIL | `datetime.strptime(date, "%Y-%m-%d")` |
| **S2** | All `time`, `time_start`, `time_end` fields match HH:MM 24-hour format | FAIL | `re.match(r"^([01]\d|2[0-3]):[0-5]\d$", time)` |
| **S3** | No two bookings/events at the same `location` have overlapping time ranges on the same `date` | WARNING | Time interval overlap detection |
| **S4** | `status` is one of: `planned`, `confirmed`, `completed`, `cancelled` | FAIL | `status in VALID_STATUSES` |
| **S5** | All `id` values are unique across all schedule records (SVC-*, EVT-*, FAC-*) | FAIL | Global ID uniqueness check |

### 5.4 Newcomers Validation Rules (N1-N5)

| Rule | Check Condition | Severity | Implementation |
|------|----------------|----------|----------------|
| **N1** | All `id` values are unique across all newcomer records | FAIL | `len(ids) == len(set(ids))` |
| **N2** | `journey_stage` is one of the 6 valid stages AND all preceding milestone stages have `completed: true` | FAIL | Ordered stage validation with milestone cross-check |
| **N3** | `first_visit` matches YYYY-MM-DD format, is a valid date, and is in the past | FAIL | `datetime.strptime(date, "%Y-%m-%d") <= today` |
| **N4** | If `referred_by` is not null, the referenced ID must exist in `members.yaml` | WARNING | Cross-file ID lookup |
| **N5** | If `settled_as_member` is not null, then `status` must be `"settled"` AND `settled_date` must be non-null | FAIL | Conditional field consistency check |

### 5.5 Bulletin Validation Rules (B1-B3)

| Rule | Check Condition | Severity | Implementation |
|------|----------------|----------|----------------|
| **B1** | Bulletin contains all required sections: `sermon`, `worship_order` (≥3 items), `announcements`, `prayer_requests` | FAIL | Section existence + minimum count check |
| **B2** | `issue_number` is strictly greater than the last entry in `generation_history` | FAIL | Monotonic increase check |

## Section 7: 3-Tier inbox/ Pipeline

### 7.1 Architecture Overview

```
inbox/
├── documents/              ← Tier A: Structured Files
│   ├── 헌금내역.xlsx          → finance.yaml
│   ├── 새신자등록카드.xlsx     → newcomers.yaml
│   ├── 심방일지.docx          → members.yaml (history)
│   └── 교인명부.csv           → members.yaml
│
├── images/                 ← Tier C: Image Files
│   ├── receipt-001.jpg        → finance.yaml (expense)
│   ├── namecard-kim.jpg       → newcomers.yaml
│   └── bulletin-text.jpg      → bulletin-data.yaml
│
└── templates/              ← Scan-and-Replicate Templates
    ├── bulletin-sample.jpg    → bulletin-template.yaml
    ├── receipt-form.jpg       → receipt-template.yaml
    └── ... (7 document types)
```

### 7.2 Tier A: Structured Files (Excel/CSV)

| Aspect | Detail |
|--------|--------|
| **Input Formats** | `.xlsx`, `.csv` |
| **Processing Method** | `openpyxl` (Excel), `pandas` (CSV) — deterministic parsing |
| **Accuracy Expectation** | **High** (95%+) — structured data with headers |
| **Error Handling** | Column mapping validation, type checking, missing value detection |
| **Human Review** | Parsed results shown to user → confirm before YAML update |
| **Failure Mode** | Preserve original file + detailed error message |
| **Example Input** | 헌금내역.xlsx with columns: 날짜, 이름, 십일조, 감사헌금, 일반헌금 |
| **Example Output** | Structured offering entries in finance.yaml |

### 7.3 Tier B: Document Files (Word/PDF)

| Aspect | Detail |
|--------|--------|
| **Input Formats** | `.docx`, `.pdf` |
| **Processing Method** | `python-docx` (Word), Claude Read tool (PDF) |
| **Accuracy Expectation** | **Medium-High** (85-90%) — semi-structured documents |
| **Error Handling** | Section boundary detection, key-value extraction with confidence |
| **Human Review** | Extracted fields shown with confidence indicators → confirm |
| **Failure Mode** | Preserve original + show raw extracted text |
| **HWP Note** | HWP (한글) is binary format — guide user to "export as PDF" |
| **Example Input** | 심방일지.docx with visit records |
| **Example Output** | History entries appended to members.yaml |

### 7.4 Tier C: Image Files (JPG/PNG)

| Aspect | Detail |
|--------|--------|
| **Input Formats** | `.jpg`, `.jpeg`, `.png` |
| **Processing Method** | Claude multimodal image analysis (built-in, no external OCR) |
| **Accuracy Expectation** | **Medium** (75-85%) — confirmation mandatory |
| **Error Handling** | Low-confidence fields marked with `[?]`, original image preserved |
| **Human Review** | **Mandatory** — all extracted data shown for verification before YAML update |
| **Failure Mode** | Show partial extraction + original image path |
| **Example Input: Receipt** | receipt-001.jpg → extract: amount, date, item, vendor |
| **Example Input: Namecard** | namecard-kim.jpg → extract: name, phone, company |
| **Example Output** | Expense entry in finance.yaml or newcomer entry in newcomers.yaml |

### 7.5 Cross-Tier Common Principles

1. **Human-in-the-loop mandatory**: All tiers require user confirmation before YAML reflection
2. **Original preservation**: Input files are never deleted or modified; moved to `inbox/processed/` after confirmation
3. **Error transparency**: Parsing failures produce detailed error messages, not silent failures
4. **Idempotency**: Re-processing the same file should not create duplicate entries (ID-based dedup)
5. **church-glossary.yaml integration**: Korean church terms in input files are recognized via glossary lookup

## Section 8: Scan-and-Replicate Catalog

### 8.1 Overview

Scan-and-Replicate allows churches to upload their existing document formats (as images or PDFs) and have the system extract the layout structure, identify fixed vs. variable areas, and generate reusable templates for automated document generation.

### 8.2 Document Type Catalog (7 Types)

#### Type 1: 주보 (Church Bulletin)

| Aspect | Detail |
|--------|--------|
| **Repetition Frequency** | Weekly |
| **Priority Tier** | Tier A (immediate) |
| **Template File** | `bulletin-template.yaml` |
| **Data Sources** | `bulletin-data.yaml` + `schedule.yaml` + `members.yaml` |
| **Fixed Areas** | Church name, logo position, section headers ("예배순서", "광고", "기도제목"), page layout |
| **Variable Areas** | Sermon title, scripture reference, worship order items, announcements content, prayer requests, birthday names, dates |
| **Output Format** | `bulletins/YYYY-MM-DD-bulletin.md` |

#### Type 2: 기부금 영수증 (Tax Donation Receipt)

| Aspect | Detail |
|--------|--------|
| **Repetition Frequency** | Annual (year-end bulk generation) |
| **Priority Tier** | Tier A (immediate) |
| **Template File** | `receipt-template.yaml` |
| **Data Sources** | `finance.yaml` + `members.yaml` |
| **Fixed Areas** | National tax form header, church registration number (사업자등록번호), legal disclaimer text, form structure (국세청 양식) |
| **Variable Areas** | Donor name (기부자 성명), resident registration number (주민등록번호), donation amount by category, donation period, issue date |
| **Output Format** | `certificates/YYYY-receipt-{member_id}.md` |

#### Type 3: 예배 순서지 (Worship Order Sheet)

| Aspect | Detail |
|--------|--------|
| **Repetition Frequency** | Weekly |
| **Priority Tier** | Tier A (immediate) |
| **Template File** | `worship-template.yaml` |
| **Data Sources** | `schedule.yaml` + `bulletin-data.yaml` |
| **Fixed Areas** | Church branding, section layout (praise, prayer, offering, sermon, benediction), formatting style |
| **Variable Areas** | Song titles, prayer leader name, sermon title/scripture, specific order items, performer names |

#### Type 4: 교회 공문 (Official Church Letter)

|--------|--------|
| **Repetition Frequency** | 2-5 per month |
| **Priority Tier** | Tier B |
| **Template File** | `letter-template.yaml` |
| **Data Sources** | `members.yaml` + `schedule.yaml` |
| **Fixed Areas** | Church letterhead, seal/stamp position (직인 위치), signature line, church address, salutation format |
| **Variable Areas** | Recipient name/organization, date, subject line, body content, reference numbers |
| **Output Format** | `certificates/YYYY-MM-DD-letter-{subject}.md` |

#### Type 5: 회의록 (Meeting Minutes)

| Aspect | Detail |
|--------|--------|
| **Repetition Frequency** | 1-4 per month |
| **Priority Tier** | Tier B |
| **Template File** | `minutes-template.yaml` |
| **Data Sources** | `members.yaml` (attendees) |

| **Output Format** | `reports/YYYY-MM-DD-minutes-{meeting_type}.md` |

#### Type 6: 세례/입교 증명서 (Baptism/Membership Certificate)

| Aspect | Detail |

### 8.3 Scan-and-Replicate Processing Pipeline

```
Step 1: Upload to inbox/templates/{category}-sample.{jpg|pdf}
    ↓
Step 2: Claude Multimodal Analysis
    ├── Extract layout structure (sections, columns, style hints)
    ├── Identify FIXED areas (church name, logo, headers, legal text)
    └── Identify VARIABLE areas (dates, names, content, amounts)
    ↓
Step 3: Generate {category}-template.yaml
    ├── fixed_elements: list of immutable content with positions
    ├── variable_slots: list of data-bound placeholders with types
    └── layout_hints: section order, alignment, spacing
    ↓
Step 4: Human Confirmation (mandatory first-time)
    ├── "Is this structure correct?"
    └── "Are all variable slots identified correctly?"
    ↓
Step 5: Repeated Auto-Generation
    └── data/*.yaml → template slots → output/{category}/{date}-{name}.md
```
